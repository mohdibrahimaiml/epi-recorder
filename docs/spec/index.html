Abstract

The EPI format defines a portable, self-contained, cryptographically sealed container for AI agent execution evidence. A .epi file is a polyglot — valid HTML for browser-based forensic inspection AND a binary envelope carrying a ZIP payload — enabling zero-install verification by regulators, auditors, and courts.

Reference Implementation: epi-recorder (Python, MIT). This specification is an open standard. The .epi file format carries no intellectual property restrictions.

1. Container Format

Envelope v2 (Polyglot)

A .epi file begins with a 128-byte binary header, followed by an HTML viewer and ZIP payload. This polyglot structure means any browser can open the file directly for forensic inspection.

ZIP Payload Marker


<!-- EPI_ZIP_PAYLOAD_START -->

All bytes between the 128-byte header and this marker = embedded viewer. All bytes after = ZIP payload.

MIME Types

Auto-detection

Read first 4 bytes: 0x3C 0x21 0x2D 0x2D → envelope-v2. 0x45 0x50 0x49 0x31 → legacy-zip. Otherwise → invalid.

4. Cryptographic Protocol

Canonical Hashing

  1. Convert model to dictionary
  2. Exclude fields: signature, governance, trust (manifest); source_type (step)
  3. Normalize UUIDs to lowercase canonical form
  4. Normalize datetimes to YYYY-MM-DDTHH:MM:SSZ (UTC, strip microseconds)
  5. JSON serialize: sort_keys=True, separators=(",", ":"), ensure_ascii=True
  6. Encode to UTF-8
  7. SHA-256 → 64-char lowercase hex

Conformance: Two implementations computing the canonical hash of identical data MUST produce identical hex strings. Test vectors provided for validation.

Manifest Signing

  1. Canonical hash of manifest (excluding signature fields)
  2. Convert hex to 32 raw bytes
  3. Ed25519 sign (RFC 8032) → 64 raw signature bytes
  4. Encode signature as 128-char hex
  5. Key ID = SHA-256(hex_public_key)[:16]
  6. Signature string: ed25519:<key_id>:<128-hex-sig>

Key ID binding: The key ID is cryptographically derived from the public key. This prevents key substitution attacks.

Manifest Verification

  1. Parse signature by splitting on : into 3 parts
  2. Reject if algorithm is not ed25519
  3. Verify key ID = SHA-256(hex_public_key)[:16]
  4. Decode 128-char hex signature to 64 raw bytes
  5. Compute canonical hash of manifest (excluding signature fields)
  6. Import Ed25519 public key (32 raw bytes)
  7. Verify signature over canonical hash bytes
  8. Return (True, "Valid") or (False, "Invalid - tampered")

Step Hash Chain

Each step in steps.jsonl contains a prev_hash field:

  • Step 0: prev_hash: null
  • Step N (N > 0): prev_hash = canonical hash of step N-1

This creates an append-only, hash-linked timeline. Any insertion, deletion, or reorder of steps breaks the chain.

6. Verification Procedure

A conformant verifier MUST perform 7 independent checks, in order:

  1. Structural — Valid envelope, extractable ZIP, required files present
  2. Integrity — Every file_manifest entry matches computed SHA-256; no extra files in ZIP (except review.json)
  3. Signature — Ed25519 signature valid against manifest canonical hash
  4. Chain — prev_hash links intact; step indices monotonic; timestamps non-decreasing
  5. Completeness — manifest.total_steps matches line count of steps.jsonl
  6. MIME — mimetype file contains exactly application/vnd.epi+zip
  7. SCITT (optional) — COSE_Sign1 statement valid, Merkle inclusion proof correct

8. Security Model

ThreatMitigation
Post-seal tamperingSHA-256 file manifest + Ed25519 signature
Evidence replayUnique workflow_id + created_at timestamp
Secret leakageHMAC-SHA256 redaction of API keys, tokens, PII
Signature spoofingKey ID cryptographically bound to public key
Step manipulationprev_hash chain breaks on insert/remove/reorder
File injectionIntegrity check rejects extra files not in manifest
Visual deceptionviewer.html hash recorded in file_manifest
Key compromiseKey revocation files in trust registry

Threats Not Mitigated

EPI is a post-execution evidence system. It records what an AI agent did — it does not prevent unauthorized access at runtime. For enforcement, use EPI_ENFORCE=1. File-level encryption is not specified by this format; implementers MAY layer transport or file-level encryption on top.

9. Conformance

An implementation is conformant if:

  1. Produces .epi files that pass the 7-pass verification procedure
  2. Produces correct canonical hashes matching the published test vectors
  3. Produces valid Ed25519 signatures that verify correctly
  4. Correctly parses and validates prev_hash step chains

Partial conformance: an implementation that does not sign manifests but otherwise produces valid files is partially conformant. Supporting only envelope-v2 (not legacy-zip) is still conformant.
